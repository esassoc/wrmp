Scales & Results

Two additions to the design system. Until now the system has dressed the exhibit experience — panels, covers, map chrome. These two jobs extend it to the WRMP topic pages: a visualization language that holds across three grains of data, and a family of components for surfacing results and resources.

Job A — Visualization templates across three data scales

One map / chart / table language, shown flexing from region-wide overviews down to a single monitoring site. A reviewer should see the same design system stretch across grains, not three unrelated treatments.

Job B — Results & Resources components

Resource cards, a Results Highlight block, two named content boxes, and a facet filter. Delivering results highlighting as reusable components is the natural job of a design system — no separate memo needed.

Two threes live on this page, and they are different axes. Data scales are about the grain of the data; audience tiers are about how deeply it is processed for the reader. They are orthogonal — any scale can be published at any tier.

Data scales (Job A)

How coarse or fine is the data?

RegionalRegion-wide datasets — habitat maps, shoreline access, demographics.
ConditionScores summarized across sites — CRAM, vegetation condition.
Site-specificDetailed data at one site — fish & fish habitat.

Audience tiers (Job B facet)

How deeply is it processed for the reader?

Tier 1Summarized information — the plain-language takeaway.
Tier 2Synthesized results — charts, comparisons, interpretation.
Tier 3Downloadable raw data — the underlying files.

Job A · Visualization templates across three data scales

Read down a column for one scale; read across a row to see the same artifact type — map, chart, or summary table — change grain. Colours, axes, legends and table chrome are shared, so the three scales read as one system. Charts use clearly-labelled representative sample data where the real dataset is not yet in-repo; the station map draws on the real data/stations.json.

Job B · Results & Resources components

The component family a WRMP topic page assembles from: resource cards keyed by type, a Results Highlight block that pulls a finding up the hierarchy, the two named content boxes the scope calls out, and a facet filter. Every colour, radius and type size resolves from tokens.css; the management-question badge reuses the shipped exhibit-tags.css.

Resource card & type badges

One card shape, one badge per resource type in the client's taxonomy. Nine types, grouped into four accent families — interactive data (teal), spatial (green), media & live (orange), reference (earth) — so the set reads as one system rather than a nine-colour rainbow. The type badge sits top-left; an audience-tier chip sits top-right.

Results Highlight box

This is Task 1.2 as a component: pull a finding up the visual hierarchy so a topic page leads with the result, not the method. It is the client's "orange box," redrawn — a warm highlight surface (no coloured left border), a big figure, the finding in a heading, and a source line. A teal variant carries a synthesis finding where there is no single headline number.

Named content boxes

The two boxes the scope names by hand — today plain "little orange boxes" on topic pages. Redrawn as a shared content-box pattern: an icon tile, a heading, one sentence, and action buttons. Each takes a distinct identity — data-teal for the finder, framework-green for the Science Framework, whose GQ colour system rides along as five swatches.
